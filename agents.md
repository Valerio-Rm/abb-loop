# 🤖 Data Migration Agent Rules (SQL Server to PostgreSQL)

## 🏗️ Project Architecture & Context
Progetto di migrazione ETL e sincronizzazione dati tra **Microsoft SQL Server** (Sorgente) e **PostgreSQL** (Target).

* **Sorgente:** MS SQL Server accessibile tramite `tds_fdw`. utilizza mcp server mssql, mssql_abb2, mssql_f200
* **Target:** PostgreSQL (Ambiente di destinazione). utilizza l'mcp server postgres_dev
* **Logging:** Sistema di logging granulare con gestione delle transazioni autonome via `dblink`.

## 🗄️ Schema Registry (PostgreSQL)
* **`application_data`**: Business logic, tabelle target e log operativi.
* **`application_audit`**: Audit trail e persistenza errori.
* **`application_staging`**: Area di transito per trasformazioni intermedie.
* **Foreign Schemas**: Schemi che mappano le tabelle SQL Server (es. `dwh_remote`, `dwh_remote2`). *Nota: il numero di questi schemi è variabile.*

## 🛠️ Mandatory Procedural Standards

### 1. Execution Window (Timezone Check)
Ogni procedura **deve** verificare la timezone del Plant prima di procedere.
* **Tabella di riferimento:** `application_data.lk_plant`.
* **Vincolo:** L'esecuzione è permessa solo se l'ora locale del Plant è tra le **00:00** e le **00:59**.
* **Logica SQL:** `v_local_hour := EXTRACT(HOUR FROM (NOW() AT TIME ZONE v_plant_timezone));`
* **Azione:** Se `v_local_hour != 0`, la procedura deve terminare immediatamente (`RETURN`).

### 2. Partition Validation
Prima di eseguire insert o elaborazioni su tabelle partizionate, verificare sempre che la partizione relativa al `plant_id` in esame esista nel database (interrogando `pg_partitions` o tramite logica applicativa).
* usa sempre la logica di seguito descritta
 BEGIN
            PERFORM application_data.create_plant_partition('**table_name**', 'day_id', **plant_ in_the_loop**);
        EXCEPTION
            WHEN OTHERS THEN
                IF SQLSTATE IN ('42P07', 'P0001') THEN
                    INSERT INTO application_data.log_operation (
                        operation_timestamp, operation_src, operation_msg, operation_caller
                    ) VALUES (
                        timezone('UTC', current_timestamp),
                        lp_procedure_name,
                        'Plant partition **table_name**' || **plant_ in_the_loop**::TEXT || ' already exists, continuing.',
                        lp_last_user
                    );
                ELSE
                    CALL application_data.log_error_write(
                        lp_procedure_name,
                        'create_plant_partition(**table_name**) failed for plant_id=' || **plant_in_the_loop**::TEXT || ': ' || SQLERRM,
                        lp_last_user
                    );
                END IF;
END;
### 3. Procedure Comment 
Dentro la procedura all'inzio prima delle dichiarazioni è necessario inserire dei commenti descrittivi in inglese di cosa fa la procedura , indicandone lo scopo. Corredare nei commenti eventuali modalità di chiamata inserendo degli esempi e descrivendone l'uso dei vari parametri

## 📝 Logging & Error Handling

### **A. Operational Logging (Standard)**
Da utilizzare per tracciare gli step interni alla procedura.
* **Tabella:** `application_data.log_operation`
* **DDL Reference:**
    ```sql
    CREATE TABLE application_data.log_operation (
        operation_ref int8 GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
        operation_timestamp timestamp DEFAULT timezone('utc'::text, CURRENT_TIMESTAMP),
        operation_src varchar(100) NOT NULL,
        operation_msg varchar(4000) NOT NULL,
        operation_caller varchar NOT NULL
    );
    ```

### **B. Error Logging (Autonomous Transaction via dblink)**
In caso di errore nel blocco `EXCEPTION`, chiamare la procedura dedicata per garantire la persistenza del log anche dopo un rollback della transazione principale.
* **Procedura:** `CALL application_data.log_error_write(p_error_src, p_error_msg, p_error_caller)`
* **Firma:** `(varchar, varchar, varchar)`
* **Meccanismo:** Utilizza `dblink_connect` e `dblink_exec` per eseguire una `INSERT` su `application_data.log_error` in una sessione separata.

## 📜 Coding Pattern Example (PL/pgSQL)

```sql
CREATE OR REPLACE PROCEDURE application_data.proc_template(p_plant_id VARCHAR)
LANGUAGE plpgsql AS $$
DECLARE
    v_proc_name TEXT := 'proc_template';
    v_tz TEXT;
    v_local_hour INT;
    v_start_ts TIMESTAMP := timezone('UTC', now());
BEGIN
    -- 1. Check Timezone
    SELECT timezone INTO v_tz FROM application_data.lk_plant WHERE plant_id = p_plant_id;
    v_local_hour := EXTRACT(HOUR FROM (now() AT TIME ZONE v_tz));
    
    IF v_local_hour != 0 THEN 
        RETURN; 
    END IF;

    -- 2. Log Start
    INSERT INTO application_data.log_operation (operation_timestamp, operation_src, operation_msg, operation_caller)
    VALUES (v_start_ts, v_proc_name, 'Inizio elaborazione', current_user);

    -- [Logica di Migrazione qui]

EXCEPTION WHEN OTHERS THEN
    -- 3. Log Error (Autonomous)
    CALL application_data.log_error_write(v_proc_name, SQLERRM, current_user);
    RAISE;
END $$;
